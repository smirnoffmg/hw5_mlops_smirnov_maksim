# Готовность ML-системы к развёртыванию в продуктивном окружении

## Из чего может состоять ML-система

Полноценная ML-система в продакшене — это не только "какая-то модель", а совокупность слоёв, каждый из которых решает свою задачу:

1. **Источники данных и data ingestion.** Производственные БД, API, CDC-стримы, файловые хранилища. Должны быть зафиксированы схемы, контракты, SLA на свежесть данных.

2. **Хранилище признаков (Feature Store).** Централизованная точка для офлайн-фичей (для обучения) и онлайн-фичей (для инференса). Решает проблему расхождения train/serve skew, обеспечивает повторное использование признаков между моделями.

3. **Версионирование данных и моделей.** DVC или аналог — чтобы по любому коммиту можно было восстановить точный набор данных и веса модели. Без этого репродуктивность невозможна.

4. **Pipeline для обучения.** Декларативный (DVC, Airflow, Kubeflow Pipelines) или императивный, с зафиксированными гиперпараметрами, логированием экспериментов в трекер (MLflow, W&B).

5. **Model Registry.** Хранилище версий моделей с метаданными, статусами (staging/production), возможностью отката к предыдущей версии.

6. **Serving.** REST/gRPC API, batch-inference job, edge-deploy. Для онлайн-сценариев — горизонтальное масштабирование, кеширование, A/B-тестирование.

7. **Контроль качества данных.** Валидация схемы, проверка на пропуски, аномалии, дрейф распределений (Great Expectations, Deepchecks, Evidently).

8. **Мониторинг.** Три уровня: инфраструктура (latency, throughput, ошибки), модель (метрики качества, drift), данные (распределения, пропуски).

9. **CI/CD.** Автоматический прогон тестов, переобучение по триггеру, deploy с canary/blue-green стратегией.

10. **Инфраструктура.** Контейнеризация (Docker), оркестрация (Kubernetes), секрет-менеджмент, IAM, наблюдаемость (Prometheus, Grafana, OpenTelemetry).

## Что из этого нужно на текущем проекте

Текущий проект — учебный, и не каждый компонент имеет смысл реализовывать. Минимум, который оправдан даже для маленькой системы:

- **Версионирование кода (Git), данных (DVC) и моделей (DVC + MLflow)** — база вопроизводимости.
- **Трекинг экспериментов (MLflow)** — иначе невозможно сравнить запуски и понять, какие изменения улучшают модель.
- **Декларативный pipeline (DVC stages)** — чтобы prepare и train запускались автоматически и пересчитывали только изменившиеся стадии.
- **Зафиксированное окружение (uv.lock + requirements.txt)** — без этого ноутбук «работает у одного, не работает у другого».

Что осознанно опущено и почему:

- **Feature Store с реальными wine-фичами** — для табличного датасета на 178 строк без временного измерения это как-то чрезмерно. Feast в проекте присутствует как демо postgres-шаблона, но не интегрирован в train.
- **Serving** — у нас нет потребителя инференса. Если бы проект шёл в прод, добавили бы FastAPI-обёртку и Docker-образ.
- **Мониторинг и Data Quality в пайплайне** — для статичного учебного датасета не имеет смысла.
- **CI/CD** — для одного разработчика и одного репо избыточно.

## Достаточно ли репозитория и блокнота Colab

Думаю, что нет.

**Версионирование кода.** Colab-блокнот — это `.ipynb`, JSON с встроенным выводом ячеек. Git показывает нечитаемые diff'ы, code review через PR-модель не работает, конфликты при параллельном редактировании практически неразрешимы. Даже если файл лежит в Git-репозитории, это не решает проблему — потому что формат принципиально не приспособлен.

**Версионирование данных.** В Colab данные обычно подгружаются через `gdown`, `wget` или монтирование Google Drive. Версии не зафиксированы — если файл на Drive обновлён, старый эксперимент уже не воспроизвести. Хеши не считаются, изменения не отслеживаются.

**Версионирование экспериментов.** Без MLflow или аналога каждый запуск ноутбука затирает предыдущий вывод. Параметры в коде, метрики в выводе ячейки, артефакт в локальной файловой системе сессии Colab, которая чистится через 12 часов простоя. Через неделю невозможно ответить на вопрос «какие гиперпараметры дали лучший результат».

**Качество данных.** Никаких проверок схемы, диапазонов, drift'а. Если в данных появились дубликаты, пропуски или NaN — узнаешь только когда модель сломается в инференсе.

Дополнительно — проблемы Colab-специфики:
- Скрытое состояние ячеек: результат зависит от порядка запусков, а не от текста кода
- Окружение Colab нестабильно: версии библиотек могут поменяться между сессиями, GPU доступен не всегда
- Нет возможности встроить блокнот в CI/CD без `nbconvert` и обвязки
- Нет API для serving — модель из ноутбука нельзя поднять как сервис без переписывания

**Вывод.** Репо + Colab — это исследовательская конфигурация, подходящая для прототипирования и demo. Для продакшена нужен минимум: Git + DVC + MLflow + контейнеризованное окружение + декларативный pipeline. Дальше — по мере роста системы добавляются Feature Store, мониторинг, CI/CD и serving.
